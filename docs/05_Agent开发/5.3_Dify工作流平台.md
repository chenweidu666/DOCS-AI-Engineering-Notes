# 1. Dify 是什么

**Dify**（社区驱动，开源仓库 **langgenius/dify**）是一套 **LLM 应用开发与运营平台**：可视化编排 **工作流 / 对话应用**、托管 **知识库与模型凭据**、通过 **API / WebApp** 等形态发布，并配套日志与基础运营能力。定位在 **低代码编排** 与 **手写 LangChain 等框架** 之间，强调「从原型到可运维」的闭环。

官方站点与文档以 [dify.ai](https://dify.ai)、[docs.dify.ai](https://docs.dify.ai) 及 [GitHub](https://github.com/langgenius/dify) 为准；功能名、节点类型随版本迭代可能调整，接入前请对照当前文档。

# 2. 核心能力

## 2.1. Workflow 与 Chatflow

- **Workflow**：偏 **单轮或批处理**，输入经节点图得到输出，适合 ETL 式生成、分类流水线等。  
- **Chatflow**：面向 **多轮对话**，可维护会话变量、历史与 Agent 多步推理。  
- 二者均支持分支、循环、变量与 **HTTP** 调用，便于产品/运营参与迭代逻辑。

## 2.2. 知识库（RAG）

内置文档分段、清洗、Embedding、向量检索与 **重排序** 等配置项；支持多知识库与引用展示。与 [4.1 RAG、Milvus 与 LangChain](../04_RAG与知识库/4.1_RAG检索增强生成.md) 理论对应：Dify 将常见链路产品化；强定制场景仍可结合 **代码节点** 或外部检索服务。

## 2.3. Agent 应用

**Agent 节点**可挂载内置工具与自定义 HTTP/OpenAPI 等，策略含 **Function Calling、ReAct** 等（具体名称与能力以当前版本为准）。适合「检索 + 多工具 + 多轮澄清」类助手；链路过长时建议拆 **子工作流** 或在外部用代码编排状态机。

## 2.4. 与框架 / 自动化工具的关系（简表）

| 维度 | Dify | LangChain / LlamaIndex 等 | Flowise / Langflow | n8n 等通用自动化 |
|------|------|---------------------------|--------------------|------------------|
| 形态 | 可视化 + 内置 LLMOps 面 | 代码优先，灵活度最高 | 偏可视化，AI 专用 | 工作流强，AI 能力需拼装 |
| RAG | 产品内一站式配置 | 自建链路 | 中等，因产品而异 | 通常非核心 |
| Agent | 画布 + 节点配置 | 完全自定义 | 因产品而异 | 多依赖插件 |
| 生产运维 | 日志、多租户等（视部署版） | 自建可观测与权限 | 因产品而异 | 成熟，但语义层需另建 |

**结论**：需要 **业务可参与、快速上线、统一模型与知识管理** 时倾向 Dify；需要 **极限定制、复杂图状态机、深度 MCP 组合** 时倾向代码栈（可与 Dify 并存：对外问答走 Dify，执行面走自研）。

# 3. 架构直觉

典型拆分：**Studio 前端** + **API / Worker 后端** + **PostgreSQL**（业务数据）+ **Redis**（队列等）+ **向量存储**（依配置选型）。支持 **官方云**、**Docker Compose 自托管** 与 **Kubernetes** 等部署；自托管需把数据库、向量引擎与备份纳入运维范围。

# 4. 工作流节点类型（常用）

| 类型 | 作用 | 提示 |
|------|------|------|
| **LLM** | 生成、分类、抽取 | 控制 max_tokens、温度与结构化输出（如 JSON） |
| **知识检索** | 向量/关键词检索 | 与 Rerank、top_k、分数阈值联动调优 |
| **HTTP 请求** | 调用外部 REST | 超时、错误体与敏感头脱敏 |
| **代码** | 轻量转换与校验 | 适合格式转换、签名，避免业务全堆在提示词里 |
| **工具 / Agent** | 多步工具调用 | 限制工具集与最大步数，控制成本与死循环风险 |
| **变量与赋值** | 会话/单次运行状态 | 多轮场景显式建模，勿依赖模型「记住」一切 |

# 5. 实战思路

## 5.1. 抽象示例：行情 / 加密市场分析类

1. **触发**：用户输入标的与周期，或定时任务驱动。  
2. **检索**：从知识库取 **指标释义、风控规则、非实时研报摘要**。  
3. **HTTP**：调用合规行情 API 取 **K 线、费率** 等（注意频率限制与条款）。  
4. **LLM**：按模板输出 **结构化结论** 与必要免责声明。  
5. **后置**：可选推送 IM/邮件；**日志**记录调用的 API 与引用片段，便于审计。

涉及 **投资建议** 的展示须合规审核；**实盘交易**应在专用风控系统执行，不宜与编排平台混为一谈。

## 5.2. Agent 类应用：推荐组合（Hybrid）

复杂助手常用 **「结构化收集 + 检索 + Agent + 输出」**：

1. 用 **表单 / 变量节点** 固化必填字段（预算、时间、约束），减少模型自由发挥。  
2. **知识检索** 提供领域事实与内部口径。  
3. **Agent 节点** 在受限工具集内做多步调用（搜索、内部 API、再检索等），并设 **最大迭代步数** 与 **会话记忆窗口**。  
4. **结束节点** 将结果格式化为 Markdown/JSON，便于 WebApp 或下游系统消费。

不必在文档中绑定某一垂直场景的长篇 API 示例；具体工具名与鉴权以各服务商文档为准。

# 6. 与 OpenClaw / 自研 Agent 对比

| 维度 | Dify | OpenClaw / 自研 Agent |
|------|------|------------------------|
| 交付形态 | 可视化 + 平台组件 | 代码优先，高度定制 |
| MCP | 视版本以插件/工具衔接 | 多 MCP Server 组合更自然 |
| 复杂状态机 | 画布 + 子流程，极复杂时可能吃力 | LangGraph 等代码编排更灵活 |
| 运维 | 自带部分运营与日志能力 | 可观测、权限需自建 |

**选型**：**快速验证、职能团队参与** 倾向 Dify；**7×24 自主任务、强编排、深度 MCP** 倾向代码栈。也可 **Dify 对外产品化，自研服务做执行与核心数据面**。

# 7. 部署与上手

**自托管（常见）**：克隆 [langgenius/dify](https://github.com/langgenius/dify)，进入仓库内 **`docker`** 目录，按官方 README 配置环境变量后执行 **`docker compose up -d`**（具体文件名与步骤以仓库当前文档为准）。

**云端试用**：使用官方云服务快速创建应用（见官网）。

**第一个应用**：新建 **Workflow 或 Chatflow** → 配置模型与凭据 → 串联 **Start → LLM（或检索）→ Answer** → Debug → 发布为 API 或 WebApp。

# 8. 典型应用场景（简表）

| 场景 | 说明 |
|------|------|
| 企业知识库问答 | FAQ + RAG，权限与知识库分区需设计 |
| 内容生成与分发 | 多节点流水线，注意版权与品牌审核 |
| 内部工具助手 | 接项目管理、代码评审提示等，工具权限最小化 |
| 多模态文档理解 | 随版本支持情况启用 OCR/图像等，以文档为准 |

# 9. 优势与局限

**优势**：上手快、协作友好、RAG/Agent **常见路径开箱**；开源可私有化；适合 AI 中台雏形与多应用复用模型配置。

**局限**：自托管依赖 **DB/向量/Redis** 等运维；极复杂控制流与性能极限调优仍可能回到代码；与各家 **最新模型特性** 之间存在产品化滞后，需关注升级说明。

# 10. 踩坑记录

- **向量库与备份**：自托管时 Postgres、Redis、向量引擎须纳入备份、扩容与升级策略。  
- **提示词与日志**：多租户场景注意 **知识库隔离** 与 **日志脱敏**，避免敏感片段外泄。  
- **Agent 死循环**：必须配置 **最大工具步数**、**单次/累计 token 上限** 与合理超时。  
- **版本升级**：大版本升级前在 **预发** 完整回归导出过的工作流与应用配置。  
- **HTTP 工具**：对外部 API 做 **速率限制、熔断与密钥轮换**，勿在画布中硬编码长期密钥。
