# 1. 为什么量化

大模型推理的主要成本来自 **显存带宽** 与 **显存容量**：FP16 权重体积大，KV Cache 与并发 batch 进一步吃满显存。将权重（及可选的激活）压到 **INT8 / INT4**，可以在相近算力下 **多放模型或多开并发**，并降低内存搬运量，从而提升 **吞吐与性价比**。量化通常以 **PTQ（训练后量化）** 为主：在少量校准数据上估计 scale / zero_point，无需完整重训，工程上最易落地。

# 2. 量化基础

## 2.1. FP32→FP16→INT8→INT4

- **FP32 / FP16**：浮点表示，表达范围大、训练与高精度推理常用；FP16 已是多数推理基线。
- **INT8**：256 个档位，配合 per-tensor 或 per-channel **scale**（及非对称时的 **zero_point**），在权重与部分激活上常用。
- **INT4**：档位更少，压缩更狠，对 **离群权重 / 敏感层** 更挑剔，往往需要 **混合精度**（部分层保留 FP16）或更稳的算法（如 AWQ 思路）。

直觉：**比特数 ↓ → 存储与带宽压力 ↓**，但数值表达能力 ↓，需用校准数据与算法换精度。

## 2.2. 对称 vs 非对称量化

- **对称量化**：零点固定在 0，只需 **scale**，硬件友好；很多权重量化默认对称。
- **非对称量化**：引入 **zero_point**，能更好贴合激活分布偏移（如 ReLU 后全为非负但不为 0 中心），表达更细但略增复杂度。

工程上常见组合：**权重量化 + 激活量化**（W8A8、W4A16 等），不同比特对应不同显存与延迟权衡。

# 3. 主流量化方案对比

| 方案 | 原理 | 精度损失 | 速度 |
|------|------|---------|------|
| GPTQ | 逐层离线量化 | 小 | 快 |
| AWQ | 激活感知权重 | 更小 | 快 |
| bitsandbytes | 在线量化 | 中 | 慢 |

选型提示：**vLLM 为主线**时，优先采用与 vLLM 部署链路对齐的压缩产物；下面第 5 节以 **llm-compressor**（vLLM 生态官方压缩库）为实践入口。

# 4. 敏感层保护（FP16 fallback）

**lm_head、部分 LayerNorm、注意力中与离群值强相关的投影层** 等对误差更敏感，全 INT4 后容易出现 **困惑度跳变或格式错乱**。常见做法是：

- 对指定层 **保持 FP16 / BF16**（混合精度量化）；
- 或在量化配置中 **排除** 若干层，由工具或配置表声明。

上线前应用 **固定评测集**（困惑度、下游任务、人工抽检生成）对比「全量化 vs 混合」再定稿。

# 5. vLLM 中量化配置实战（配合 llm-compressor）

**llm-compressor**（[vllm-project/llm-compressor](https://github.com/vllm-project/llm-compressor)）是 **与 Transformers 兼容** 的压缩库，面向 **优化后在 vLLM 中部署** 的 LLM。典型路径是：**原始 HF 模型 → 离线压缩（INT8/INT4 等）→ 产出兼容权重与配置 → vLLM 按量化格式加载 serving**。

建议按官方 README 锁定 **Python / PyTorch / vLLM** 版本矩阵。一般步骤可概括为：

1. **准备校准样本**：数百～数千条与线上一致的 prompt（或中间层激活统计所需数据），分布越贴近生产越好。
2. **离线运行压缩**：使用仓库提供的 recipe / 脚本，选择目标方案（如权重量化比特、是否 W4A16 等），导出压缩后的模型目录。
3. **vLLM 加载**：使用与压缩格式匹配的启动参数（`quantization`、模型路径等，以当前 vLLM 文档为准）；先做 **单卡冒烟**，再压 **并发与延迟**。

不必手写底层 GEMM：重点是把 **校准数据、敏感层策略、评测门禁** 做稳；具体 API 以官方示例为准，避免复制过期函数名。

# 6. 实测：太初项目量化收益

- **显存**：−50%+
- **吞吐**：↑明显
- **精度**：可接受（以业务评测集与线上抽样为准）

# 7. 踩坑记录

- **校准集与线上一致性**：分布漂移会导致量化参数「偏科」，离线指标好、线上差。
- **工具与 vLLM 版本错配**：压缩产物格式与推理端支持的 kernel 不一致时，启动失败或默默降级。
- **忽略敏感层**：首训全 INT4 不验 lm_head 等，易出现灾难性劣化。
- **只盯平均 loss**：长尾 query、工具调用 JSON、代码块等需单独看 bad case。
