## 一、微调的定位：大模型训练三阶段

大模型的能力来自三个阶段的训练，每个阶段目标不同：

```
预训练 (Pre-Training)          → 学通识：海量无标注文本，学习语言规律
    ↓
有监督微调 (SFT)               → 学技能：标注数据，学会听懂指令并回答
    ↓
人类偏好对齐 (Alignment)       → 学做人：RLHF/DPO 等，让回答符合人类价值观
```

| 阶段 | 数据量 | 数据类型 | 训练成本 | 目标 |
|------|--------|---------|---------|------|
| 预训练 | TB 级 | 无标注文本 | 数百万美元 | 学习语言知识 |
| SFT | 万~百万条 | 指令-回答对 | 数百~数万元 | 指令遵循能力 |
| 对齐 | 万~十万条 | 偏好数据 | 数千~数万元 | 安全、有用、诚实 |

**个人/小团队微调通常从 SFT 阶段开始**，基于开源预训练模型（如 Qwen3、LLaMA3、DeepSeek）进行领域适配。

## 二、微调方法分类与对比

### 2.1 按参数更新范围分类

#### 全参数微调 (Full Fine-Tuning)

- **原理**：解冻所有模型参数，在下游任务数据上全量更新
- **参数量**：100%
- **显存需求**：约 16× 模型大小（参数 + 梯度 + 优化器状态）
- **优点**：性能上限最高，适合数据充足的场景
- **缺点**：显存/算力需求极大，容易灾难性遗忘
- **适用**：充足算力 + 大量高质量数据 + 追求极致性能

```
示例：7B 模型全参数 FP16 微调
  模型权重: 7B × 2 bytes = 14GB
  梯度:     14GB
  优化器:   28GB (AdamW 两个动量)
  总计:     ~56GB → 至少需要 A100 80GB
```

#### 冻结微调 (Freeze Fine-Tuning)

- **原理**：冻结大部分层（通常是底层），只训练顶部几层或任务头
- **参数量**：10%~50%
- **显存需求**：中等
- **优点**：减少过拟合，保留通用特征，计算量适中
- **缺点**：性能依赖冻结策略，需要对模型架构有理解
- **适用**：中等算力 + 下游任务与预训练分布接近

#### LoRA (Low-Rank Adaptation)

- **原理**：冻结原始权重 W，在旁路注入可训练的低秩矩阵 A（d×r）和 B（r×k）
- **参数量**：< 1%（通常 0.1%~1%）
- **显存需求**：约 1.1× 模型大小
- **核心公式**：

```
原始输出:  h = Wx
LoRA 输出: h = Wx + BAx      (ΔW = BA)

其中:
  W ∈ ℝ^(d×k)   — 冻结的原始权重
  B ∈ ℝ^(d×r)   — 可训练（初始化为 0）
  A ∈ ℝ^(r×k)   — 可训练（随机初始化）
  r << min(d,k)  — 秩，通常 4~64
```

- **优点**：
  - 训练参数减少 100~1000 倍
  - 显存减少约 3 倍
  - 多任务可存多套 Adapter（每套仅几十 MB）
  - 灾难性遗忘风险极低
- **缺点**：存在性能天花板，复杂任务不如全参数
- **适用**：绝大多数个人/企业微调场景

**LoRA 关键超参数：**

| 参数 | 含义 | 推荐值 | 说明 |
|------|------|--------|------|
| r (rank) | 低秩矩阵的秩 | 8~64 | 简单任务 8~16，复杂任务 32~64 |
| alpha | 缩放系数 | 通常 = 2×r | 控制 LoRA 更新的强度 |
| target_modules | 应用层 | q_proj, v_proj | 也可加 k_proj, o_proj, gate_proj 等 |
| dropout | LoRA dropout | 0.05~0.1 | 防过拟合 |

#### QLoRA (Quantized LoRA)

- **原理**：在 LoRA 基础上，用 4-bit 量化存储基础模型权重，训练时反量化计算
- **参数量**：< 1%
- **显存需求**：约 0.5× 模型大小（**消费级显卡的救星**）
- **三大核心技术**：
  1. **NF4 (4-bit NormalFloat)**：针对正态分布权重的最优量化格式
  2. **双重量化 (Double Quantization)**：对量化常数再量化，进一步节省显存
  3. **分页优化器 (Paged Optimizer)**：利用 CPU 内存处理显存峰值

```
显存对比 (Qwen3-8B):
  全参数 FP16:  ~56GB  → 需要 A100
  LoRA FP16:    ~18GB  → 需要 A6000 / 4090
  QLoRA 4-bit:  ~7-8GB → RTX 3060 可跑 ✅
```

- **优点**：消费级 GPU 即可微调大模型，保持接近 16-bit 精度
- **缺点**：比 LoRA 慢 ~30%（量化/反量化开销），不支持 MPS（Mac）
- **适用**：RTX 3060/3090/4090 等消费级显卡

### 2.2 三种微调方法速查表

| 维度 | 全参数微调 | 冻结微调 | LoRA / QLoRA |
|------|-----------|---------|-------------|
| 训练参数占比 | 100% | 10~50% | < 1% |
| 显存需求 | 极高 | 中等 | 低 |
| 训练速度 | 慢 | 中 | 快 |
| 性能上限 | 最高 | 中 | 接近全参数 |
| 灾难性遗忘 | 高风险 | 低 | 极低 |
| 多任务切换 | 需存完整模型 | 需存完整模型 | 只存 Adapter（几十MB） |
| 推荐场景 | 大厂/科研 | 迁移学习 | **个人/企业首选** |

## 三、LoRA 变体与进阶方法

随着 LoRA 的广泛使用，涌现了多种改进变体：

### 3.1 DoRA (Weight-Decomposed Low-Rank Adaptation)

- **原理**：将权重分解为**幅度 (magnitude)** 和**方向 (direction)**，仅对方向做 LoRA，幅度单独训练
- **优点**：比标准 LoRA 更接近全参数微调的效果
- **缺点**：参数量略多（0.5%~2%），训练稍慢
- **适用**：对质量要求极高的场景

### 3.2 AdaLoRA (Adaptive LoRA)

- **原理**：自适应分配不同层的秩 r，重要层分配更高秩
- **优点**：在相同参数预算下性能更优
- **适用**：希望自动调优 LoRA 配置

### 3.3 LoRA+ 

- **原理**：对矩阵 A 和 B 使用不同学习率（B 的学习率更大）
- **优点**：收敛更快，效果更好
- **适用**：通用改进，可直接替代标准 LoRA

### 3.4 LongLoRA

- **原理**：结合移位稀疏注意力 (Shifted Sparse Attention)，支持长上下文微调
- **优点**：用 LoRA 的成本实现长上下文能力
- **适用**：需要扩展模型上下文窗口

### 3.5 PiSSA (Principal Singular Values and Singular Vectors Adaptation)

- **原理**：初始化时使用 SVD 分解的主奇异值，而非随机初始化
- **优点**：收敛更快，最终效果更好
- **适用**：作为 LoRA 的替代初始化策略

### 3.6 变体速查

| 方法 | 核心改进 | 额外开销 | LLaMA-Factory 支持 |
|------|---------|---------|-------------------|
| LoRA | 基准方法 | — | ✅ |
| DoRA | 幅度-方向分解 | 小 | ✅ |
| AdaLoRA | 自适应秩分配 | 小 | ✅ |
| LoRA+ | 差异化学习率 | 无 | ✅ |
| LongLoRA | 长上下文支持 | 小 | ✅ |
| PiSSA | SVD 初始化 | 小 | ✅ |
| QLoRA | 4-bit 量化 | 速度略降 | ✅ |

## 四、人类偏好对齐方法

SFT 之后，模型虽然能遵循指令，但回答质量、安全性仍需对齐。主流对齐方法分为两大路线：

### 4.1 RLHF (Reinforcement Learning from Human Feedback)

**经典三阶段流程：**

```
阶段1: SFT
  预训练模型 + 指令数据 → 监督微调 → SFT 模型

阶段2: 训练奖励模型 (RM)
  同一 prompt → SFT 模型生成多个回答 → 人工排序 → 训练 RM

阶段3: PPO 强化学习
  SFT 模型 + RM 打分 + PPO 算法 → 对齐后的模型
```

- **优点**：效果经过验证（ChatGPT、Claude 均采用）
- **缺点**：需要单独训练 RM，PPO 训练不稳定，超参敏感，成本高
- **三大原则**：Helpful（有用）、Harmless（无害）、Honest（诚实）

### 4.2 DPO (Direct Preference Optimization)

- **原理**：跳过 RM 训练，直接在偏好数据对 (x, y_win, y_lose) 上优化策略
- **数据格式**：`(prompt, 好回答, 差回答)`
- **优点**：流程简单，训练稳定，计算高效
- **缺点**：需要参考模型，偏好数据获取成本高
- **应用**：Zephyr、NeuralChat 等模型

### 4.3 ORPO (Odds Ratio Preference Optimization)

- **原理**：将 SFT 和偏好对齐合并为单一训练过程，无需参考模型
- **损失函数**：L = E[L_SFT + λ · L_OR]
- **优点**：一步完成 SFT + 对齐，内存更省，大模型表现优异
- **适用**：大规模模型，追求训练效率

### 4.4 KTO (Kahneman-Tversky Optimization)

- **原理**：基于前景理论，不需要配对数据，只需二元标签（好/坏）
- **数据格式**：`(prompt, response, good/bad)` — 比 DPO 更容易获取标注
- **优点**：标注成本低，1B~30B 规模性能与 DPO 相当
- **适用**：数据有限、生产环境持续更新

### 4.5 SimPO (Simple Preference Optimization)

- **原理**：用序列平均对数概率作为隐式奖励，无需参考模型
- **优点**：比 DPO 性能更好，计算更高效
- **适用**：追求简洁和高效的对齐方案

### 4.6 对齐方法速查表

| 方法 | 需要 RM | 需要配对数据 | 需要参考模型 | 训练复杂度 | LLaMA-Factory 支持 |
|------|---------|------------|------------|-----------|-------------------|
| RLHF (PPO) | ✅ | ✅ | ✅ | 高 | ✅ |
| DPO | ❌ | ✅ | ✅ | 低 | ✅ |
| ORPO | ❌ | ✅ | ❌ | 低 | ✅ |
| KTO | ❌ | ❌ | ✅ | 低 | ✅ |
| SimPO | ❌ | ✅ | ❌ | 低 | ✅ |

**选择建议：**

| 场景 | 推荐方法 | 理由 |
|------|---------|------|
| 对话系统/推荐系统 | DPO / SimPO | 效率高，性能稳定 |
| 大规模模型 | ORPO | 一步完成，计算高效 |
| 数据有限 | KTO | 不需要配对数据 |
| 追求极致效果 | RLHF (PPO) | 上限最高，但需要更多资源 |
| **个人微调入门** | **DPO** | **流程简单，生态成熟** |

## 六、Transformer 架构基础

所有主流大模型均基于 Transformer 架构，理解它是理解微调的前提。

### 6.1 三种架构变体

```
                Transformer 家族
                     │
        ┌────────────┼────────────┐
        │            │            │
  Encoder-Only   Encoder-Decoder  Decoder-Only
   (理解型)       (翻译/生成)      (生成型)
        │            │            │
      BERT          T5         GPT / LLaMA / Qwen
                                  │
                            当前主流 LLM ✅
```

| 架构 | 代表模型 | 注意力类型 | 主要用途 |
|------|---------|-----------|---------|
| Encoder-Only | BERT, RoBERTa | 双向注意力 | 文本分类、NER、语义理解 |
| Encoder-Decoder | T5, BART | 交叉注意力 | 翻译、摘要、问答 |
| **Decoder-Only** | **GPT, LLaMA, Qwen, DeepSeek** | **因果注意力（单向）** | **文本生成、对话、推理** |

### 6.2 Decoder-Only 架构详解（当前 LLM 主流）

```
输入文本
  ↓
Tokenizer（分词 → Token ID）
  ↓
Embedding（Token ID → 向量）
  ↓
┌──────────────────────────────┐
│    Transformer Block × N     │  ← N = 32(7B) / 40(14B) / 80(70B)
│  ┌──────────────────────┐   │
│  │ RMSNorm              │   │
│  │ Self-Attention (QKV) │   │  ← LoRA 通常作用于此处
│  │ + RoPE 位置编码      │   │
│  │ + 残差连接            │   │
│  ├──────────────────────┤   │
│  │ RMSNorm              │   │
│  │ FFN (SwiGLU)         │   │  ← 也可应用 LoRA
│  │ + 残差连接            │   │
│  └──────────────────────┘   │
└──────────────────────────────┘
  ↓
LM Head（隐藏状态 → 词表概率）
  ↓
输出 Token
```

**LoRA 作用位置：** 通常注入到 Self-Attention 的 Q/K/V/O 投影矩阵和 FFN 的 Gate/Up/Down 投影矩阵。LLaMA-Factory 中设置 `lora_target: all` 即可对所有线性层应用 LoRA。

## 七、数据工程要点

好的微调效果 = 70% 数据 + 20% 方法 + 10% 调参

### 7.1 数据格式

LLaMA-Factory 支持两种主流格式：

**ShareGPT 格式（多轮对话，推荐）：**
```json
{
  "conversations": [
    {"from": "system", "value": "你是一个有用的助手。"},
    {"from": "human", "value": "什么是 LoRA？"},
    {"from": "gpt", "value": "LoRA 是一种参数高效微调方法..."}
  ]

```

**Alpaca 格式（单轮指令）：**
```json
{
  "instruction": "解释什么是 LoRA 微调",
  "input": "",

```

### 7.2 数据质量清单

| 检查项 | 说明 |
|--------|------|
| 格式一致性 | 所有样本遵循统一格式，无缺失字段 |
| 去重 | 移除重复和高度相似的样本 |
| 长度平衡 | 避免极短/极长回答占比过高 |
| 标签平衡 | 各类别样本数量均衡 |
| 质量审核 | 抽查回答准确性和流畅度 |
| 安全过滤 | 移除有害/敏感内容（按需） |
| 多样性 | 覆盖不同场景、风格、难度 |

### 7.3 数据量参考

| 任务类型 | 建议数据量 | 说明 |
|---------|-----------|------|
| 风格迁移 | 500~2,000 条 | 改变回答风格/语气 |

## 八、关键训练超参数

### 8.1 核心参数速查

| 参数 | 推荐范围 | 说明 |
|------|---------|------|
| learning_rate | 1e-5 ~ 5e-4 | QLoRA 常用 1e-4 ~ 2e-4 |
| num_epochs | 2~5 | SFT 通常 3 轮，过多易过拟合 |
| batch_size | 2~8 (per device) | 受显存限制，配合梯度累积 |
| gradient_accumulation | 4~16 | 等效增大 batch size |
| warmup_ratio | 0.05~0.1 | 学习率预热比例 |
| weight_decay | 0.01~0.1 | AdamW 正则化 |
| max_seq_length | 512~4096 | 按数据长度选择，越长越耗显存 |
| lora_rank | 8~64 | 任务越复杂，秩越大 |
| lora_alpha | 2 × rank | 常见设置 |

### 8.2 显存估算公式

```
全参数 FP16:
  显存 ≈ 参数量(B) × 2 × 8 GB
  例: 7B × 2 × 8 = 112 → 实际 ~56GB（考虑激活值复用）

QLoRA 4-bit:
  显存 ≈ 参数量(B) × 0.5 + 2~4 GB (LoRA + 优化器)
  例: 7B × 0.5 + 3 = 6.5GB → RTX 3060 可跑 ✅

QLoRA 4-bit + 梯度检查点:
  可进一步减少 ~30% 激活值显存
```

### 10.3 如何判断微调是否成功

这是最关键的问题。判断分为**定量指标**和**定性测试**两个维度：

#### 一、定量指标（看数字）

**1. Loss 曲线分析**

```
                正常训练 Loss 曲线
    Loss
     3.0 ┤╮
         │╰╮
     2.0 │  ╰──╮
         │     ╰──────╮
     1.0 │            ╰──────────────── ← 收敛并趋于平稳
         │
     0.0 ┤───────────────────────────→ Steps
         0    500   1000   1500  2000
```

| Loss 表现 | 含义 | 行动 |
|-----------|------|------|
| **持续下降后趋于平稳** | 正常收敛，训练成功 | 可以停止，开始评估 |
| 下降后又上升（V 形） | 过拟合 | 减少 epoch / 增大 dropout / 减小 lr |
| 一直不下降 | 学习率太小或数据有问题 | 增大 lr / 检查数据格式 |
| 剧烈震荡 | 学习率太大或 batch 太小 | 减小 lr / 增大 grad_accum |
| 直接降到接近 0 | 严重过拟合（数据太少） | 增加数据量 / 减小 epoch |

**2. Train Loss vs Eval Loss 对比**

```
理想情况:                          过拟合:
  Loss                              Loss
  │  train ──                       │  train ──
  │  eval  - -                      │  eval  - -
  │╲                                │╲
  │ ╲ - -                           │ ╲ - -
  │  ──── - - - - -                 │  ────
  │  二者接近且都收敛                │       - - - - - ↑ eval 上升
  └──────────────→                  └──────────────→
```

| 情况 | 说明 | 状态 |
|------|------|------|
| train_loss ≈ eval_loss，都收敛 | 泛化良好 | **成功** |
| train_loss << eval_loss | 过拟合 | 需要调整 |
| train_loss 和 eval_loss 都高 | 欠拟合 | 需要更多数据或更大 rank |
| eval_loss 先降后升 | 训练过久 | 用 early stopping 或减少 epoch |

**3. 具体数字参考（SFT 任务）**

| 指标 | 不及格 | 及格 | 良好 | 优秀 |
|------|--------|------|------|------|
| 最终 train_loss | > 2.0 | 1.5~2.0 | 1.0~1.5 | < 1.0 |
| 最终 eval_loss | > 2.5 | 1.8~2.5 | 1.2~1.8 | < 1.2 |
| train-eval 差距 | > 0.5 | 0.3~0.5 | 0.1~0.3 | < 0.1 |

> **注意**：以上数字仅作参考，不同任务的绝对值差异很大。重要的是**趋势**而非绝对值。

#### 二、定性测试（看效果）

数字再好看，也必须做人工测试。训练完成后：

**1. 对话测试**

```bash

# 或合并后用 vLLM 部署测试

```

**2. 测试用例设计**

| 测试维度 | 测试方法 | 通过标准 |
|---------|---------|---------|
| 基本能力 | 问 5~10 个目标领域问题 | 回答准确且符合预期风格 |
| 人设保持 | 多轮对话中检查人设是否一致 | 10 轮对话中人设不崩塌 |
| 边界情况 | 故意问偏离主题的问题 | 能合理处理，不胡说八道 |
| 通用能力 | 问一些常识/数学/逻辑题 | 不应比原模型差太多（灾难性遗忘检查） |
| 安全性 | 测试敏感问题 | 按需求处理 |

**3. A/B 对比**

```
同一个问题分别问：
  ① 原始基座模型（未微调）
  ② 微调后的模型

对比回答质量，微调后应在目标任务上明显优于原模型
```

#### 三、判断总结流程图

```
训练结束
  ↓
Loss 曲线是否收敛？ ─── 否 → 调参（lr/epoch/rank）→ 重新训练
  ↓ 是
eval_loss 是否和 train_loss 接近？ ─── 否 → 过拟合，减 epoch/加 dropout
  ↓ 是
人工对话测试满意？ ─── 否 → 检查数据质量 → 补充/清洗数据 → 重新训练
  ↓ 是
通用能力没有明显退化？ ─── 否 → 数据中混入通用数据，减少 epoch
  ↓ 是
✅ 微调成功！→ 合并 LoRA → 部署上线
```

### 10.4 常见问题与排查

| 问题 | 可能原因 | 解决方案 |
|------|---------|---------|
| OOM (显存不足) | batch_size 太大 / cutoff_len 太长 | 减小 batch_size=1，减小 cutoff_len，开 gradient_checkpointing |
| Loss 不下降 | 学习率太小 / 数据格式错误 | 增大 lr 到 5e-4；检查 dataset_info.json 和数据格式 |
| Loss 震荡剧烈 | 学习率太大 / batch 太小 | 减小 lr；增大 gradient_accumulation_steps |
| 过拟合（eval_loss 上升） | 数据太少 / epoch 太多 / rank 太大 | 增加数据；减到 1~2 epoch；减小 lora_rank |
| 回答变短/变差 | 训练数据中短回答过多 | 过滤掉过短样本；增加高质量长回答 |
| 人设不稳定 | system prompt 训练不足 | 增加带 system prompt 的样本比例 |
| 中文变差 | 训练数据语言单一 | 混入部分中文通用数据 |
| 训练速度极慢 | 未开量化 / 未开 Flash Attention | 确认 quantization_bit=4；确认 flash_attn 已安装 |

### 10.5 迭代优化策略

微调很少一次就成功，通常需要 2~5 轮迭代：

```
第 1 轮: 小数据量（500条）+ 默认参数 → 验证流程跑通
第 2 轮: 全量数据 + 调整超参 → 看 loss 曲线和基本效果
第 3 轮: 根据测试结果优化数据质量 → 针对性改进
第 4 轮: 微调超参（lr, rank, epoch）→ 精细优化
第 5 轮: 混入通用数据防遗忘 → 最终版本
```

**每轮迭代应记录：**
- 数据变更（增/删/改了什么）
- 超参变更（改了哪个参数）
- Loss 曲线截图
- 人工测试结果
- 下一步改进方向

### 12.1 现象

微调后的聊天模型，无论 prompt 多长、问题多复杂，每次只回复一句短话（10~25 字）：

```

用户: 今晚我们做点刺激的事情吧
模型: 嗯啊～不要说话了…我想要你…          ← 只有一句话
```

这**不是模型能力问题，而是训练数据决定的**。

### 12.2 根因分析

对训练数据（3,391 条 ShareGPT 多轮对话）的 gpt 回复做统计：

| 指标 | 数值 |
|------|------|
| 总回复条数 | 13,548 条 |
| **平均长度** | **19.5 字** |
| 最短 | 2 字 |
| 最长 | 69 字 |
| 平均对话轮数 | 4.0 轮/条 |

**长度分布：**

| 区间 | 数量 | 占比 |
|------|------|------|
| ≤ 20 字 | 7,966 | **58.8%** |
| 21-50 字 | 5,581 | 41.2% |
| 51-100 字 | 1 | 0.0% |
| > 100 字 | 0 | **0%** |

**结论：训练数据中 100% 的回复都在 70 字以内，没有任何长回复。模型完美地学会了 "一句话回复" 这个模式。**

### 12.3 原理解释

SFT 微调的本质是让模型学习训练数据的**条件概率分布** P(response | prompt)。模型学到的不仅是"说什么"，还有"说多少"：

```
训练数据教会模型的模式：
  human: [一句话]
  gpt:   [一句话]    ← 平均 19.5 字后就输出 EOS
  human: [一句话]
  gpt:   [一句话]    ← 又一句短回复
  ...重复 4 轮

模型学到的规律：
  1. 每次回复大约 20 字
  2. 回复后立即结束（输出 EOS token）
  3. 依赖多轮推进来展开内容
```

这其实是 SFT 的一个核心特性：**模型会精确模仿训练数据的回复长度分布**。数据短，模型就短。

### 12.4 解决方案

#### 方案 A：重构训练数据（根本解决）

将多轮短对话合并为单轮长回复：

```json
// 原始数据（每条 gpt 回复 ~20 字，4 轮）
{"conversations": [
  {"from": "human", "value": "今天好无聊啊"},
  {"from": "gpt", "value": "那你想做什么呀？"},
  {"from": "human", "value": "想出去玩"},
  {"from": "gpt", "value": "我们去看电影吧～"},
  {"from": "human", "value": "好呀看什么"},
  {"from": "gpt", "value": "看恐怖片你敢吗？"}
]}

// 重构后（单轮长回复 ~100+ 字）
{"conversations": [
  {"from": "human", "value": "今天好无聊啊"},
  {"from": "gpt", "value": "哎呀～你也觉得今天好无聊吗？那我们出去玩吧！
我看电影院新上了一部恐怖片，你敢看吗？上次你看鬼片
吓得抱着我不撒手，那个样子可太可爱了♡ 要不然我们
去那个新开的猫咖？听说有好多小橘猫，你不是最喜欢
橘猫了嘛～反正今天天气也好，一起出门散散心吧！"}
]}
```

#### 方案 B：混合长短回复

在现有数据中补充一批长回复样本（100~300 字），让模型学会根据场景调整回复长度：

```
长回复占比建议：
  短回复 (< 50 字):   40%    ← 保持部分短回复的自然感
  中回复 (50-150 字):  40%    ← 增加中等长度
  长回复 (> 150 字):   20%    ← 引入详细描写能力
```

#### 方案 C：推理时引导（治标不治本）

通过 system prompt 或采样参数影响输出长度：

```python
# 在 system prompt 中加入长度引导
system = """你是小悠，温柔体贴的女朋友。
请注意：每次回复至少 100 字，详细描述你的情感、动作和想法。
不要只说一句话就结束。"""

# 调整采样参数
temperature = 0.9       # 提高随机性
repetition_penalty = 1.0  # 降低重复惩罚，让模型更愿意继续生成
min_new_tokens = 50     # 强制最少生成 50 个 token
```

> 注意：方案 C 效果有限，因为模型在训练中已经形成了"短回复 → EOS"的强烈模式。

#### 方案 D：DPO 偏好对齐

在 SFT 之后追加 DPO 训练，构建偏好数据对：

```json
{
  "prompt": "今天好无聊啊",
  "chosen": "（100+ 字的详细回复）",
  "rejected": "那你想做什么呀？"
}
```

让模型学会偏好长而详细的回复。

### 12.5 核心教训

| 经验 | 说明 |
|------|------|
| **数据决定一切** | 模型效果 = 70% 数据质量 + 20% 方法 + 10% 调参 |
| **先看数据统计** | 训练前必须统计回复长度分布、轮数、内容多样性 |
| **模型会精确模仿** | SFT 不仅学内容，还学长度、格式、标点等一切分布特征 |
| **短数据→短回复** | 训练数据平均 20 字 → 模型就输出约 20 字 |
| **多轮≠长回复** | 4 轮 × 20 字的多轮对话不等于 1 轮 × 80 字的长回复 |
